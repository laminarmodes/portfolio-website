Mobile-first data visualization

I am SwiftUI addicted. Learning Swift was a great experience and I was super excited to transition to SwiftUI and continue to explore one of my favorite parts of coding which is data visualization.

I love, Love, LOVE making data charts in SwiftUI (and will soon do the same in Jetpack Compose)

This app is still in the making so I am not ready to reveal what it does but I am very keen to share the journey I have taken, from ideation to proof-of-concept app and soon the app store.

Getting ideas out

I tend to come up with ridiculous ideas on a daily basis. Thankfully I keep most of them to myself but every now and then I try to break them down into the basic steps. Here, the user can accomplish three things 1) add information quickly, 2) sort information comfortably and 3) present information clearly.

What problem does the UI solve?

People need diverse options to organize, communicate & interpret information: charts, words, symbols, images, tables, lists. I imagined different ways to toggle between all options in a simple UI. The mobile-first approach is space limiting so smart design patterns are needed to simplify the UI, without losing information.

Any excuse to write some beautiful code

Data Binding is super simple to implement in SwiftUI and give me the space to focus on creating a complex UI that lets the user explore, drill-down, compare, scan and browse. The actual user input (shown later) is persisted, but in a simple way for now in order to collect user feedback and iterate on. Once the demonstrator is validated I can complete the core data design for the final app.

Simplifying Data Summaries

In the bar chart, each bar height represents the number of items that are contained inside each large summary card. The color of the bar then corresponds to the color of the large summary card. Color-blind friendly color palettes should be used to label data sets.

An additional interaction could be, when the user selects a bar, it automatically scrolls to the corresponding card for them.

Taking a mobile-first design approach to data visualization is something I have always been interested in because it could really free the analyst from their desk and allow them to explore data, monitor trends, do comparisons and listen out for alerts while moving around.

Keeping Drill-Downs Simple

This view would allow the user to drill down past the summary, and look at all the information contained within all summary cards at once. Each row should correspond to the color of the large card.

Each 'row of cards' corresponds to a single line on the line graph and each 'individual card' corresponds to a 'single point' on the line. Each project can have a different overall color theme; in this case, a warm color-blind friendly color ramp.

I remember a statement from a very famous Decision Scientist who described the data analyst as, the one who "holds the flashlight". I see the mobile-first data visualization approach as giving them a really efficient portable flashlight.

Show the details, only when they need them

Looking now at the small cards, clicking one card will reveal the details of the card and alternatively, the user can view a row of cards in a list view which may be preferred for presentations and discussions.

Now this might be a good time to also mention that this design is only in Swift and was not originally drawn in a design software such as AdobeXD or Figma. I however still encourage using design tools at the earliest possible because they will always be the best standard when it comes to creating and testing different designs, as well as documentation.

User input

New cards can be added to each individual row (see 'Test Item, Test Description') and a new category can be added as an empty list (see 'Test List'). In this proof-of-concept app, I used Core Data to persist the information that the user entered.

Transition animations

However when a new card is added, the transition can be sudden and the card may go unnoticed. The card just kind of appears out of nowhere. It is important to remember to use animations for these interactions and the beauty of SwiftUI is that you can implement them with just a few lines of code.

What's next

I am looking foward to sharing more details on what this app does once I get closer to publishing on the app store. For now, I just hope to share my journy and inspire other UI designers to get into SwiftUI.
